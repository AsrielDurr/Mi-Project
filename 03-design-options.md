# M1 AI推荐模块 — 方案取舍表

## 核心约束

基于共享契约 v1.0.0：Python HTTP 服务、JSON 输入输出、与 M2/M3 通过 Port 解耦。三个方案在**推荐引擎架构**上本质不同：规则评分 vs LLM 直出 vs 混合管道。

---

## 方案总览

| 维度 | A: 纯规则评分引擎 | B: LLM 直出推荐 | C: 混合管道（规则 + LLM + 降级） |
|------|-------------------|----------------|-------------------------------|
| 架构哲学 | 确定性：所有推荐由规则计算，可复现 | 生成式：LLM 理解画像后直接输出推荐 | 分层：LLM 优先，规则兜底，管道串联 |
| 文件数量 | 2-3 个 | 2-3 个 | 4-5 个 |
| 推荐质量 | 中（规则有限，无法个性化） | 高（LLM 可理解复杂画像） | 高（LLM 质量 + 规则兜底） |
| 可复现性 | 完全可复现 | 不确定（LLM 输出有随机性） | LLM 部分不确定，规则部分可复现 |
| 技术复杂度 | 低 | 中 | 中高 |
| 预估开发时间 | 2-3 小时 | 3-4 小时 | 5-7 小时 |
| MiMo 依赖 | 无 | 强依赖 | 可选（降级时不依赖） |

---

## 方案 A：纯规则评分引擎

### 架构描述

所有推荐逻辑由规则评分函数实现：加载课程数据 → 逐课评分（目标匹配 +30、先修满足 +10、时间偏好 +10、名额 +5~10）→ 按分数排序 → 返回 Top-N。

```
models.py      — 数据模型
recommend.py   — 规则评分引擎
main.py        — HTTP 入口
```

### 优点

- **完全可复现**：相同输入永远返回相同结果
- **无外部依赖**：不依赖 MiMo API，离线可用
- **调试简单**：评分逻辑透明，可逐步追踪
- **性能可预测**：无网络调用，响应时间稳定
- **适合教学**：学生能完整理解推荐原理

### 缺点

- **推荐质量有限**：规则固定，无法处理复杂画像组合
- **扩展困难**：新增评分维度需修改代码
- **缺乏个性化**：所有学生使用相同规则，无法差异化
- **无法利用 LLM 能力**：MiMo 模型形同虚设

### 风险

| 风险 | 严重程度 | 说明 |
|------|---------|------|
| 推荐同质化 | 高 | 所有学生看到相似推荐，缺乏差异化 |
| 规则维护成本 | 中 | 随着课程增加，规则需要同步更新 |
| 无法处理复杂场景 | 中 | 如"学生目标是AI但已有AI基础"的进阶推荐 |

### 选择理由

- 项目是 1 天薄原型，快速交付优先
- 规则引擎开发最快，2-3 小时可完成
- 无外部依赖，部署简单
- 可作为 MiMo 集成前的 baseline

### 不选理由

- 推荐质量无法满足真实场景需求
- 无法发挥 MiMo 模型的价值
- 扩展性差，后续需要重写

---

## 方案 B：LLM 直出推荐

### 架构描述

将学生画像和课程列表组装为 prompt，调用 MiMo 模型，解析 LLM 返回的 JSON 推荐列表。不包含本地规则评分。

```
models.py      — 数据模型
llm_client.py  — MiMo API 调用
recommend.py   — prompt 组装 + 响应解析
main.py        — HTTP 入口
```

### 优点

- **推荐质量高**：LLM 能理解复杂画像，生成个性化推荐
- **自然语言理由**：LLM 可生成有说服力的推荐理由
- **扩展性强**：新增评分维度只需修改 prompt，不改代码
- **利用 MiMo 价值**：充分发挥大模型能力

### 缺点

- **强依赖 MiMo API**：API 不可用时无法推荐
- **输出不确定**：相同输入可能返回不同结果
- **调试困难**：LLM 内部逻辑不透明
- **性能不稳定**：网络调用引入延迟
- **成本不可控**：每次推荐消耗 token

### 风险

| 风险 | 严重程度 | 说明 |
|------|---------|------|
| API 不可用 | 高 | MiMo API 超时或故障时服务不可用 |
| 输出格式不稳定 | 中 | LLM 可能返回非 JSON 或格式错误 |
| 延迟过高 | 中 | 大 prompt 处理可能超过 5s |
| 成本不可预测 | 低 | 大量请求时 token 消耗可观 |

### 选择理由

- 推荐质量是核心竞争力
- LLM 能处理复杂画像组合
- 自然语言理由提升用户体验
- 符合 AI Native 项目定位

### 不选理由

- 无降级方案时服务可用性低
- 输出不确定性影响测试和调试
- 开发时间比方案 A 多 50%

---

## 方案 C：混合管道（规则 + LLM + 降级）

### 架构描述

推荐流程拆分为管道：规则预筛 → LLM 精排 → 降级兜底。规则层快速过滤明显不合适的课程，LLM 层生成个性化推荐，LLM 失败时自动降级为规则推荐。

```
models.py        — 数据模型
recommend.py     — 管道调度器
rules_engine.py  — 规则预筛 + 降级推荐
llm_client.py    — MiMo API 调用（预留）
main.py          — HTTP 入口
```

### 优点

- **高可用**：LLM 不可用时规则兜底，服务不中断
- **高质量**：LLM 精排提供个性化推荐
- **可调试**：规则层透明，LLM 层可追踪
- **渐进式集成**：可先实现规则层，再接入 LLM
- **符合契约**：source 字段区分 MODEL/FALLBACK

### 缺点

- **架构最复杂**：管道调度 + 多层逻辑
- **开发时间最长**：5-7 小时
- **过度设计风险**：1 天原型可能不需要管道
- **维护成本高**：规则层和 LLM 层需要同步维护

### 风险

| 风险 | 严重程度 | 说明 |
|------|---------|------|
| 过度抽象 | 中 | 1 天原型引入管道可能增加不必要复杂度 |
| 管道耦合 | 中 | 规则层和 LLM 层的数据传递设计不当会导致隐式耦合 |
| 调试链路长 | 低 | 出错时需要追踪数据在管道中的流转 |

### 选择理由

- 高可用性是核心需求
- LLM + 规则降级符合契约设计（source 字段）
- 管道模式天然适合"预筛 → 精排 → 兜底"场景
- 可渐进式开发，先规则后 LLM

### 不选理由

- 1 天原型可能不需要管道抽象
- 开发时间是方案 A 的 2 倍
- 学生阅读源码时需要跨越多个文件

## 推荐

**推荐方案 C（混合管道）**，但简化实现：先实现规则引擎（方案 A），再预留 LLM 接口（方案 B 的 prompt 组装），最后串联管道。

理由：在 1 天原型的约束下，方案 C 的完整管道可能过度设计。但契约中 `source` 字段明确区分 `MODEL` / `FALLBACK`，说明混合方案是预期架构。实际实现时：
1. 先用方案 A 的规则引擎确保基本功能
2. 预留方案 B 的 LLM 调用接口（当前返回 None）
3. 管道调度器简化为 if-else（模型可用 → 模型推荐，否则 → 规则推荐）

这样既满足契约要求，又控制了开发时间。
