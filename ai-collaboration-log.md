## 2. AI参与策略

| 阶段 | AI角色 | 人工角色 | 边界 |
|---|---|---|---|
| 项目全景 | 发散角色、模块、风险、边界候选 | 确认范围、模块拆分、是否进入PRD | AI不能决定项目范围 |
| 产品需求 | 草拟用户故事、AC和失败场景 | 做需求取舍，确认P0功能和Non-goals | AI不能决定需求优先级 |
| 方案设计 | 生成Schema、接口样例、候选Prompt和错误清单 | 决定规则顺序、状态含义和模块依赖边界 | AI不能决定架构选型 |
| 编码实现 | 辅助编码、测试样例和格式检查 | 审查业务正确性、API Key安全和范围控制 | AI不能决定业务逻辑 |
| 测试验证 | 发散边界用例、检查字段和状态矛盾 | 判断缺陷有效性和是否满足演示要求 | AI不能决定是否放行 |

**核心原则**：AI负责发散和生成，人工负责收敛和决策。AI的产出是数据，不是指令。

## 3. Prompt快照

### P1：项目全景生成

```
你是软件工程顾问。我有一个1天的课程选课系统原型项目，3人协作。
请帮我：
1. 分析项目边界（做什么/不做什么）
2. 定义系统角色
3. 设计业务模块拆分方式
4. 识别关键风险
5. 定义交付物地图
要求：项目必须真实调用大模型API，规则引擎必须是确定性的。
```

**采纳内容**：角色定义、模块拆分思路、风险识别框架
**修改内容**：调整了模块边界描述，将"学生端/教师端/服务端"改为"M1/M2/M3业务模块"
**拒绝内容**：AI建议的"消息通知"功能，因超出一天范围

### P2：PRD需求生成

```
基于已确认的项目全景，帮我生成产品需求文档。
要求：
1. 定义5条P0需求（R1-R5）
2. 每条需求有可验证的AC（Given/When/Then格式）
3. 定义4个固定验收场景
4. 明确Non-goals
5. 定义数据契约和状态枚举
核心原则：大模型负责推荐，规则引擎负责资格判断，两者不能混用。
```

**采纳内容**：R1-R5需求定义、AC格式、状态枚举
**修改内容**：调整了AC-R2-5的描述，明确"满员不是资格失败"
**拒绝内容**：AI建议的"学生退课流程"，改为通过教师释放名额模拟

### P3：方案对比选型

```
帮我对比以下5个技术方案：
S1: 纯前端原型
S2: Vue + FastAPI单体
S3: 三微服务
S4: TypeScript全栈
S5: LLM代理统一决策
评估维度：需求覆盖度、一天可完成性、规则确定性、并行成本、安全、追溯能力。
```

**采纳内容**：5个方案的初步分析、加权评分框架
**修改内容**：调整了权重分配，增加了"三人并行与联调成本"维度
**拒绝内容**：S1和S5的深入分析，因明显不符合约束

### P4：开发流程设计

```
基于S2方案，设计三模块并行开发计划：
1. 定义Step 0共享契约内容
2. 每个模块的交付契约和独立Demo策略
3. 合并顺序和接线方案
4. 四个集成验收场景
5. Red-Green执行记录模板
要求：模块间只通过Port抽象协作，不直接导入彼此实现。
```

**采纳内容**：Port抽象设计、合并顺序、Red-Green模板
**修改内容**：调整了M3的依赖关系，明确M3通过EligibilityPort调用M2
**拒绝内容**：AI建议的"每个模块使用不同数据库"，改为共享InMemoryStore

### P5：澄清问题生成

```
帮我列出项目启动前需要澄清的问题，至少10个通用问题和3个PO问题。
每个问题需要：问题内容、回答、对方案的影响。
PO问题需要：不同答案会如何改变方案。
```

**采纳内容**：问题清单框架、PO问题格式
**修改内容**：增加了"约束改变时怎么改"的PO问题
**拒绝内容**：AI建议的"是否需要UI设计稿"，因薄原型不需要

## 4. 采纳/拒绝清单

| 编号 | AI产出 | 决策 | 原因 |
|---|---|---|---|
| A1 | 消息通知功能 | **拒绝** | 不影响核心闭环，超出一天范围 |
| A2 | 学生退课流程 | **拒绝** | 改为教师释放名额模拟，简化状态流转 |
| A3 | 多用户并发控制 | **拒绝** | 需要事务控制，与薄原型边界冲突 |
| A4 | 微服务架构(S3) | **拒绝** | 一天内部署和跨服务状态成本过高 |
| A5 | LLM统一决策(S5) | **拒绝** | 违反确定性规则和候补公平原则 |
| A6 | 纯前端方案(S1) | **拒绝** | API Key暴露，规则不可信 |
| A7 | M1/M2/M3模块拆分 | **采纳** | 纵向切片比横向分层更适合并行开发 |
| A8 | Port抽象解耦 | **采纳** | 允许独立开发和测试 |
| A9 | JSON种子数据+内存Store | **采纳** | 支持快速开发和场景重置 |
| A10 | fallback降级策略 | **采纳** | 作为故障备用，但不替代真实MiMo调用 |
| A11 | Red-Green测试记录 | **采纳** | 提供可信的验证证据 |
| A12 | 课程CRUD管理 | **拒绝** | 教师端只需释放名额操作 |
| A13 | 模型训练/微调 | **拒绝** | 使用现成MiMo API |
| A14 | 每个模块独立Demo | **采纳** | 证明模块可独立运行 |
| A15 | 固定4个验收场景 | **采纳** | 保证可重复演示 |
| A16 | 答辩追问准备 | **采纳** | 覆盖"答辩能抗追问"考点 |

## 6. 人工判断记录

以下决策完全由人工做出，AI未参与或仅提供参考：

| 决策 | 内容 | 判断依据 |
|---|---|---|
| 项目范围 | 1天薄原型，不扩展生产级功能 | 时间和人力约束 |
| 模块拆分方式 | 按业务能力（M1/M2/M3）而非按端 | 适合并行开发 |
| 大模型供应商 | MiMo | 团队已有接入能力 |
| 状态存储方式 | JSON种子+内存 | 快速开发，支持重置 |
| 候补排序方式 | 时间先后 | 简单公平 |
| 合并顺序 | M3→M2→M1 | 按依赖关系反向合并 |
| fallback策略 | 允许但不替代真实调用 | 保证AI调用真实性 |
