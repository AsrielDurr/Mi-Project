# AI Log — AI课程选课冲突与候补调整系统

> 本日志记录项目两天开发周期中具有实际决策或交付影响的AI协作。每条记录包含目的、输入、建议、人工判断和验证五个字段。日志覆盖需求、方案、分工、契约、M3开发、集成、验证、界面修正和交付整理；纯操作性事务不纳入记录，也不展开M1、M2的具体实现过程。

## 第1条：题目理解与薄原型范围收敛

| 字段 | 内容 |
|------|------|
| **目的** | 从课程题目中识别真正需要证明的判断链路，并把三人两天开发范围收敛为可以运行和答辩的最小原型。 |
| **输入** | 提供了课程题目“AI课程选课冲突与候补调整系统”的场景页、业务闭环页和最低原型要求。材料明确要求读取学生目标、推荐课程、检查规则、管理选课与候补，并至少演示一条主路径、一次核心判断、一个边界或失败场景和可追溯结果；同时设定三人协作、原型做薄、不建设完整教务系统的边界。 |
| **建议** | 建议把项目核心收敛为“推荐—规则—状态—候补重算—追溯”，保留学生输入、AI推荐、重复/先修/冲突检查、三种选课结果、教师释放演示名额、候补递补和Trace；将权限、正式数据库、并发抢课、完整课程管理和复杂审批列为Non-goals。 |

| **验证** | `01_project-flow-map.md`和`02_product-prd.md`均明确记录项目目标、P0、Non-goals和V1—V4；当前集成系统能够运行推荐、规则、候补和追溯主路径，边界与代码实现一致。 |

## 第2条：推荐、资格与候补三个概念的澄清

| 字段 | 内容 |
|------|------|
| **目的** | 避免把“推荐合适”“当前有资格”和“候补排名靠前”混为同一个结论，明确系统中的核心业务判断。 |
| **输入** | 提供了题目中的两个起始线索：两门推荐课程都可能符合学习目标，但上课时间完全重叠；候补第一名出现空位后也可能因为后来选择了冲突课程而不能补入。边界条件是AI可以提供建议，但公平、资格和状态变化必须可重复验证。 |
| **建议** | 建议将推荐结果与资格判断彻底分开：AI只回答“适合学什么”，规则引擎回答“当前能不能选”，候补服务回答“状态变化后按什么顺序处理”。满员不应判为规则失败，而应在规则通过后进入候补；候补补入前必须重新执行资格检查。 |
| **人工判断** | 采纳了三层判断模型，并要求文档说明“推荐不等于可选，候补顺序不等于必然补入”。没有采纳让模型统一输出最终选课结论的方向，因为这会使同一输入产生不稳定结果，也无法证明候补公平。 |
| **验证** | PRD将`RuleDecision`、`EnrollmentStatus`和`WaitlistStatus`分成三组枚举；V3验证规则`PASS`后因满员进入`WAITLISTED`，V4验证第一名失效后被`SKIPPED`并继续处理第二名。 |

## 第3条：五个技术方案的发散与最终选型

| 字段 | 内容 |
|------|------|
| **目的** | 在编码前比较不同架构路线，选择与两天工期、三人能力和题目考点相匹配的技术方案。 |
| **输入** | 提供了三人团队、需要真实大模型、前后端至少两端、固定模拟数据、允许内存状态和必须可追溯等约束，并要求给出五条总体方案供人工选择。比较不能只看性能和开发成本，还要考虑项目适配度、团队技能、状态一致性和演示稳定性。 |
| **建议** | 发散出五条路线：纯前端直连模型、Vue+FastAPI单体、模型/规则/状态三个微服务、TypeScript全栈单体、LLM统一裁决。建议保留S2、S3、S4深入比较，以需求覆盖、两天完成可能性、AI与规则边界、状态一致、团队技能和演示稳定性进行加权，最终推荐S2“Vue双角色前端+FastAPI单体后端+JSON种子+内存Store”。 |
| **人工判断** | 选择S2作为最终方案。拒绝S1，因为浏览器直连会暴露Key且双端容易形成两套状态；拒绝S5，因为AI直接裁决资格违反公平和确定性；没有选择S3，因为两天内处理多个服务的部署、通信和一致性风险过高；S4虽可行，但团队后端实现更适合Python。 |
| **验证** | `03_design-options.md`保留5→3→1的比较和最终决策；实际项目只需启动一个FastAPI和一个Vite进程，三个业务模块在单体内部隔离并共用状态，证明S2已按预期落地。 |

## 第4条：MiMo、内存状态与fallback边界确认

| 字段 | 内容 |
|------|------|
| **目的** | 确定大模型供应商、运行状态方式和模型失败后的演示策略，同时防止降级结果被误当成真实AI证据。 |
| **输入** | 提供了最终方案S2，并人工指定大模型供应商为MiMo；同时说明可以接受服务重启后状态丢失，但对“是否允许fallback进入正式演示”和“页面为何必须标记降级推荐”需要进一步解释。边界是系统必须真实接入模型，但现场网络或模型失败时不能让整个演示中断。 |
| **建议** | 建议使用MiMo Adapter和环境变量配置，状态采用JSON种子+内存Store，通过reset恢复场景。fallback可以作为故障备用，但必须在接口、页面和Trace中明确标记来源；真实MiMo主流程至少需要一份成功调用证据，fallback不能代替该证据。 |
| **人工判断** | 采纳MiMo、内存Store和有条件fallback。接受进程重启后状态丢失，换取两天内不引入数据库迁移和事务；对fallback进行了限制性采纳：允许保障演示连续，但页面必须显示“降级推荐”，答辩时不能把它称为模型成功。目前决定后续补充真实MiMo截图。 |
| **验证** | 推荐页面区分`MODEL`与`FALLBACK`；后端具有结构校验、重试和验证脚本；README记录安全配置方法；`05_test-strategy.md`和`06_qa-gates.md`将真实MiMo截图保留为唯一未关闭的AI真实性证据。 |

## 第5条：团队分工从按端拆分改为纵向模块

| 字段 | 内容 |
|------|------|
| **目的** | 解决“学生端、教师端、服务端”分工看似直观但工作量不平衡、成员之间等待依赖过强的问题。 |
| **输入** | 初始讨论提出一人负责学生内容、一人负责教师内容、一人负责服务端，并要求检查三人工作量是否相当。随后进一步设定约束：不必强行按学生端、教师端、服务端拆分；三个成员应能只根据文档独立完成各自模块，最终合并后能够集成运行。 |
| **建议** | 最初评估按端分工可用于表达角色页面，但服务端会承担模型、规则、状态和追溯的大部分核心工作。建议改为三个纵向业务模块：M1 AI推荐、M2规则与选课、M3候补重算与追溯；每个模块都包含自身需要的前端、后端、测试和Fake依赖，并通过共享Ports最终接线。 |
| **人工判断** | 修改了最初的按端方案，正式采用M1/M2/M3纵向切片。人工判断的主要理由不是形式上的平均，而是降低合并风险：如果只按端拆分，两个前端成员无法独立验证完整判断，服务端成员又成为所有人的阻塞点。学生端和教师端最终保留为使用角色视图，不再对应人员职责。 |
| **验证** | 三个远程分支分别形成模块交付，`04_dev-workflow.md`记录模块所有权、Ports和接线方式；最终组合入口能够同时运行学生端和教师端，而`01_project-flow-map.md`已清除旧的按端人员分工。 |

## 第6条：共享契约与可合并开发规则

| 字段 | 内容 |
|------|------|
| **目的** | 让三个成员在不同位置开发时，只依赖同一份接口和状态定义，减少合并时的字段漂移。 |
| **输入** | 提供了三模块并行开发要求，并询问“共享契约是不是GitHub”。进一步要求先生成共享目录，并强调后续开发必须与共享契约保持一致，不能为了某个模块改动契约涉及的内容。 |
| **建议** | 解释GitHub只是共享和版本管理载体，共享契约本身应由OpenAPI、领域Schema、枚举、前端事件和合法响应示例组成。建议先冻结`project/contracts/`，规定模块只依赖Ports，不直接导入其他模块实现；接口不一致时优先修改实现，契约变化需要三方确认。 |
| **人工判断** | 采纳“契约先行、实现服从契约”的方式，并明确要求M3开发和后续集成不得修改共享契约。没有把GitHub分支本身当作契约，因为分支只能保存文件，不能自动保证三个模块对字段和状态的理解一致。 |
| **验证** | `project/contracts/`包含OpenAPI、Schema、枚举、事件和示例；M3具有共享契约一致性测试；该阶段的8个业务接口存在冻结契约测试。后续为支持界面课程和学生切换，契约1.1.0正式加入两个只读目录接口，并将契约测试扩展为10个接口。 |

## 第12条：本地运行与MiMo配置说明整理

| 字段 | 内容 |
|------|------|
| **目的** | 让项目能够在Windows PowerShell和Conda环境中被其他成员复现，并降低MiMo Key配置错误和泄露风险。 |
