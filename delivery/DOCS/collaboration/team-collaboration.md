# 团队分工、集成与问题记录

## 1. 分工原则

团队没有按学生端、教师端和服务端分工，而是按三个可以独立开发和验证的业务模块拆分：

| 模块 | 分支 | 负责范围 | 当前结果 |
|---|---|---|---|
| M1 AI推荐 | `xuzhu` | MiMo、结构校验、重试、fallback、推荐组件和测试 | 已合并 |
| M2 规则选课 | `WangJX` | 重复、先修、冲突、容量分流、选课组件和测试 | 已合并 |
| M3 候补追溯 | `hhw` | Store、候补重算、Trace、教师组件和测试 | 已合并 |
| 集成收尾 | 集成分支，最终进入`main` | 单Store接线、V1—V4、README、课程切换和候补统计 | 已完成 |

这种分法避免服务端成员承担全部核心逻辑，也使每个模块能够以共享Ports和可控替身独立测试。

## 2. 共享协作边界

- `project/contracts/`是唯一共享契约。
- 模块不直接导入其他模块实现。
- 接口不一致时优先修改实现，不私改契约。
- 正式入口只允许一个Store和Trace来源。
- Fake只用于模块独立测试，不能进入最终组合入口。

## 4. 主要问题与处理

| 问题 | 处理方式 | 实际影响 |
|---|---|---|
| 按端分工导致服务端工作量过重 | 改为M1/M2/M3纵向模块 | 每个成员交付自身闭环和测试 |
| 各分支目录和契约副本不同 | 集成时只迁入模块实现，统一使用冻结契约 | 避免字段和状态漂移 |
| M3独立Demo使用Fake资格判断 | 正式入口注入M2真实RuleEngine | V4能够证明真实跨模块检查 |
| 学生提交候补后教师端需要看到变化 | 正式入口使用同一个Store | V3跨模块状态可见 |
| 教师端只能看AI201 | 增加8门课程切换 | 可查看全部种子课程 |
| 补入后候补人数没有减少 | 只统计`WAITING`，历史记录单列 | 页面数量与状态含义一致 |
| MiMo配置步骤复杂 | README增加安全输入和Key类型判断 | 降低启动错误和Key泄露风险 |

## 5. 实际执行偏差

- 原计划曾考虑按学生端、教师端和服务端分配人员，最终未采用。
- M1、M2原始分支并非完全使用最终项目目录，集成时进行了结构适配。
- M3独立模式的Fake只用于模块验证，集成后已替换。
- 教师课程切换和候补统计是集成运行后发现并补充的收尾问题。
