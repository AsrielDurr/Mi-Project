# 决策备忘与最终建议

## 1. 最终建议

采用已经落地的S2方案：Vue双角色前端、FastAPI单体后端、M1/M2/M3模块隔离、JSON种子数据、单一内存Store和统一Trace。

当前项目不需要重新拆分架构。后续工作应集中在真实MiMo证据、演示材料和少量边界测试，而不是增加生产级功能。

## 2. 关键决策

| 决策 | 结论 | 影响 |
|---|---|---|
| 人员分工 | M1推荐、M2规则选课、M3候补追溯 | 三人可独立开发并按契约合并 |
| AI边界 | 推荐和解释 | 资格、容量、候补公平由代码负责 |
| 模型 | MiMo `mimo-v2.5-pro` | 后端环境变量配置，真实截图待补 |
| 状态存储 | 内存Store | 接受重启丢失，使用reset恢复 |
| fallback | 保留但明确标记 | 不能冒充真实模型成功 |
| 满员处理 | 规则PASS后进入候补 | 区分资格结论和业务状态 |
| 候补策略 | 原顺序逐人重检 | 第一名失效后继续下一名 |
| 名额变化 | 容量加1模拟 | 不实现完整退课状态机 |
| 最终技术结论 | GO WITH WARN | 可进入演示准备，真实MiMo证据待关闭 |

## 3. 决策依据

- 两天内需要稳定启动和重复演示。
- 三人需要并行开发并能独立验证模块。
- AI和规则边界必须清晰。
- 学生端、教师端和候补服务必须看到同一状态。
- 候补公平需要确定性规则和Trace证据。
- 生产级数据库、权限和并发不属于本轮目标。

## 4. 未采用的方向

- 纯前端：Key和状态可信度不足。
- 微服务：两天内集成和部署成本过高。
- LLM统一裁决：不可重复、不可审计且影响公平。
- 按学生端、教师端、服务端分工：工作量失衡且容易形成串行依赖。

## 5. 约束变化时的处理

- 要求持久化：用数据库实现现有Store Ports。
- 要求并发抢课：增加事务、条件更新、唯一约束和并发测试。
- 要求真实退课：容量不变、已选人数减1，增加退课状态和Trace。
- 要求AI解释规则：允许解释M2结果，但不能修改结论。
- 更换模型供应商：替换LLM Adapter，保持推荐服务接口。

任何重大变化应先更新PRD和共享契约，再补测试，最后修改实现。
