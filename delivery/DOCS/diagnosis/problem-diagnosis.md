# 问题诊断、澄清与范围

## 1. 真实问题

学生选择课程需要同时满足个人学习目标和学校硬约束。课程适合学生，不代表当前可以选上；候补顺序靠前，也不代表名额出现时仍然具备资格。

本项目的核心矛盾有三项：

1. 自然语言目标与固定课程目录之间需要可解释的匹配。
2. 推荐结果与重复、先修、冲突和容量等硬规则必须分开。
3. 候补期间状态会变化，系统必须重新检查并保持处理顺序公平。

因此项目目标不是只做推荐页面，而是形成“推荐—规则—状态—候补重算—追溯”的完整闭环。

## 2. 进一步确认的业务事实

- 满员不是规则失败，而是规则通过后的容量分流。
- 第一名只代表处理顺序，不代表当前资格。
- 学生端和教师端必须共享同一个状态来源。
- fallback必须显示来源，否则会混淆真实模型和降级结果。
- 固定演示需要reset，而不是反复手改JSON。
- “释放一个名额”是原型中的容量变化模拟，不是完整退课。

## 4. 项目目标

- 提供目录内、可解释的课程推荐。
- 使用确定性代码执行重复、先修和时间冲突检查。
- 支持选课成功、候补和拒绝三种状态。
- 状态变化后重新检查候补资格。
- 第一名失效后继续处理下一名。
- 通过Trace说明推荐、规则和状态变化来源。
- 固定场景能够reset后重复演示。

## 5. Non-goals

- 真实教务系统和真实学生数据。
- 登录、权限和账号管理。
- 数据库持久化和灾难恢复。
- 多用户并发抢课。
- 完整课程管理和真实退课。
- 复杂审批、消息通知和生产部署。
- 模型训练或微调。
