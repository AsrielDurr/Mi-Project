# 原型演示流程

## 1. 演示前准备

1. 确认后端运行在`127.0.0.1:8000`。
2. 确认前端运行在`127.0.0.1:5173`。
3. 配置MiMo环境变量；如本次演示真实模型，先运行验证脚本。
4. 打开学生端和教师端页面。
5. 在教师端执行reset，恢复固定场景。

## 2. 主路径

1. 学生端输入学习目标。
2. 点击“获取AI推荐”。
3. 指出页面展示的推荐来源、模型、理由和不确定性。
4. 选择一门推荐课程。
5. 展示M2逐条规则检查。
6. 展示最终选课状态和`trace_id`。

说明重点：AI负责推荐，最终资格由规则代码判断。

## 3. 冲突或候补路径

可根据现场数据选择其中一个：

- 冲突课程：展示`REJECTED`和关联课程，说明规则失败不会写入状态。
- 满员课程：展示`WAITLISTED`和候补排名，说明满员不是资格失败。

## 4. 教师候补路径

1. 打开教师端。
2. 选择“人工智能基础（AI201）”。
3. 展示容量、等待候补和已处理记录。
4. 点击“释放一个名额”。
5. 点击“重新计算候补”。
6. 展示第一名因时间冲突被`SKIPPED`。
7. 展示第二名被`PROMOTED`，等待人数随之减少。
8. 展示Trace中的释放、检查、跳过和补入事件。

## 5. 失败恢复

- 状态被前一次演示污染：执行reset。
- MiMo暂时不可用：展示明确的“降级推荐”，同时说明真实模型证据另行保存。
- 前端暂时无法启动：使用FastAPI `/docs`演示接口和Trace。

## 6. 演示结论

本原型证明的不是一个完整教务系统，而是AI推荐、确定性规则、候补动态调整和决策追溯之间的最小可信闭环。
