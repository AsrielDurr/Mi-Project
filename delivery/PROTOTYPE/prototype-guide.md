# 原型说明、运行方式与边界

## 1. 原型目的

本原型用于证明以下闭环可以运行：

```text
自然语言目标
→ MiMo课程推荐
→ 确定性规则检查
→ 选课成功、候补或拒绝
→ 教师释放演示名额
→ 候补资格重检与顺序递补
→ 决策追溯
```

## 2. 页面组成

### 学生端

- 输入学习目标。
- 查看MiMo推荐或明确标记的降级推荐。
- 查看推荐分数、理由和不确定性。
- 选择课程并提交选课。
- 查看重复、先修和时间冲突的逐条结果。
- 查看`ENROLLED`、`WAITLISTED`或`REJECTED`状态及追溯编号。

### 教师端

- 在8门种子课程之间切换。
- 查看已选/容量、可用名额、等待候补和已处理人数。
- 查看当前等待队列和已处理记录。
- 释放一个演示名额。
- 触发候补重新计算。
- 查看补入、跳过原因和Trace事件。
- reset恢复固定演示场景。

## 3. 判断边界

MiMo负责理解学习目标、推荐目录内课程并解释理由。重复、先修、时间冲突、容量分流、候补顺序和状态更新均由确定性代码完成。

推荐不等于有资格选课；候补第一名也不等于一定获得名额。候补学生在等待期间状态可能变化，因此补入前必须重新检查。

## 4. 运行方式

完整环境变量和安装命令见：

```text
Day5/integration-workspace/project/README.md
```

运行时需要两个终端，分别启动FastAPI和Vite。未配置MiMo Key时系统仍能以明确标记的fallback运行，但这种结果不能作为真实MiMo成功证据。

## 5. 固定演示场景

| 场景 | 预期结果 |
|---|---|
| V1 正常推荐并选课 | 推荐目录内课程，规则通过，最终`ENROLLED` |
| V2 时间冲突 | 返回`BLOCK/REJECTED`，不写入选课或候补状态 |
| V3 满员进入候补 | 规则`PASS`，业务状态`WAITLISTED/WAITING` |
| V4 第一名失效 | 第一名`SKIPPED`，第二名`PROMOTED/ENROLLED` |

## 6. 当前边界

- 没有登录、权限和真实身份。
- 没有正式数据库，状态在进程重启后丢失。
- 不支持多人并发抢最后名额。
- “释放一个名额”通过容量加1模拟，不是完整退课。
- 不支持课程改时后批量处理所有受影响学生。
- 不支持例外审批、消息通知和生产部署。
- 学生端部分位置仍以课程ID为主要标识。
- 真实MiMo成功截图尚待补充。
