## 2. 真正要解决的问题

学生选课不只是"选一门课"这么简单。当学生带着学习目标和偏好来选课时，会遇到一系列**硬约束**：

- **先修约束**：没学过A课就不能选B课
- **时间约束**：两门课同一时间只能选一门
- **重复约束**：已选的课不能再选
- **容量约束**：课满了就进候补

核心问题是：**当AI推荐的课程因为这些硬约束无法选上时，系统如何公平、透明地处理？**

具体表现为：
1. AI推荐了一门好课，但学生时间冲突 —— 怎么办？
2. AI推荐了一门好课，资格也够，但课已满 —— 怎么办？
3. 候补第一名等了很久，但等到空位时已经选了冲突课 —— 怎么办？
4. 教师释放了一个名额，谁该获得？依据是什么？

---

## 3. 核心冲突

| 冲突 | 表现 | 本质 |
|---|---|---|
| 推荐 ≠ 可选 | AI说"这门课适合你"，但规则说"你没资格" | AI负责理解需求，规则负责判断资格，两者职责不同 |
| 候补靠前 ≠ 必得名额 | 候补第一名，但等空位时可能已经失去资格 | 候补顺序是静态的，资格是动态的 |
| 一天时间 vs 完整系统 | 要演示完整闭环，但只有1天 | 必须做极致取舍，只保留核心判断路径 |
| 三模块并行 vs 依赖关系 | M3需要M2的规则，M1输出需要M2处理 | 必须通过Port抽象解耦，允许Fake/Stub独立运行 |

## 4. 利益相关方

| 角色 | 关注点 | 决策权 |
|---|---|---|
| 学生 | 获得适合自己的课程推荐，公平参与选课 | 选择课程、查看结果 |
| 教师/教务 | 课程正常运行，候补公平递补 | 释放名额、触发重算 |
| 开发团队 | 1天内交付可演示的薄原型 | 技术选型、范围取舍 |
| 评委/答辩老师 | 看到真实AI调用、确定性规则、可追溯证据 | 判断项目是否达标 |

---

## 5. 约束条件

| 约束 | 具体内容 | 对方案的影响 |
|---|---|---|
| 时间 | 1天（约6-8小时有效开发时间） | 排除微服务、正式数据库、复杂动画 |
| 人数 | 3人协作 | 必须并行开发，模块需要解耦 |
| 大模型 | 必须真实调用MiMo | 需要适配器、校验、fallback |
| 数据 | 模拟数据，不接真实教务 | JSON种子数据+内存Store |
| 部署 | 本地运行，不部署到服务器 | 前端Vite开发服务器 + FastAPI |
| 前端 | Vue 3 + TypeScript | 快速完成表单、卡片、状态展示 |

---

## 6. 假设

| 编号 | 假设 | 如果不成立的后果 |
|---|---|---|
| A1 | MiMo API在演示期间可用 | 需要fallback降级，但必须标记为非正常结果 |
| A2 | 1天内3人可以完成3个模块的并行开发 | 需要削减非核心功能，只保留主路径 |
| A3 | 模拟数据足够覆盖所有验收场景 | 需要精心设计6-8门课程和3名学生的数据 |
| A4 | FastAPI + Vue 3是团队最熟悉的技术栈 | 学习成本会占用开发时间 |
| A5 | 评委关注AI调用真实性而非UI精美度 | UI可以简洁，但AI调用和规则判断必须真实 |

---

## 7. 非目标（明确不做）

| 不做的内容 | 原因 | 如果做了的代价 |
|---|---|---|
| 注册/登录/权限 | 与核心选课冲突无关 | 增加3-4小时，偏离主路径 |
| 真实教务数据 | 需要学校配合，不可控 | 无法保证数据可用性 |
| 多用户并发 | 需要事务控制，超出薄原型范围 | 增加5-8小时，引入复杂性 |
| 完整课程CRUD | 教师端只需释放名额 | 增加3-4小时，偏离核心场景 |
| 学生退课流程 | 本轮用教师释放名额模拟 | 简化状态流转 |
| 消息通知 | 不影响核心闭环 | 增加2-3小时 |
| 生产级部署 | 薄原型本地运行即可 | 增加部署和运维成本 |
| 模型训练/微调 | 使用现成MiMo API | 偏离应用开发主题 |

## 8. 成功标准

| 标准 | 可验证方式 |
|---|---|
| 真实AI调用 | 服务端实际调用MiMo API，返回结构化推荐，追溯记录包含模型名称 |
| 确定性规则 | 同一输入得到同一结果，规则不由模型决定 |
| 三种选课结果 | ENROLLED、WAITLISTED、REJECTED均可演示 |
| 候补动态调整 | 教师释放名额后，第一名失效、第二名补入 |
| 决策可追溯 | 每次操作有trace_id，可查看完整决策链 |
| 四场景可重复 | reset后重新执行，结果一致 |
| 证据可信 | Red-Green记录、AI协作日志、分工记录都落文件 |

## 9. 问题全貌思维导图

```text
学生选课遇到冲突和容量限制
├── 核心问题：AI推荐 ≠ 有资格选
│   ├── 推荐适合但时间冲突 → 规则拒绝
│   ├── 推荐适合但课已满 → 进入候补
│   └── 推荐适合且有资格 → 选课成功
├── 候补问题：顺序 ≠ 必得名额
│   ├── 第一名等待期间选了冲突课 → 失去资格
│   ├── 第二名仍然合格 → 补入
│   └── 名额用完 → 停止递补
├── 约束：1天、3人、薄原型
│   ├── 必须并行开发 → Port抽象 + Fake/Stub
│   ├── 必须真实AI调用 → MiMo适配器 + fallback
│   └── 必须可重复演示 → 固定数据 + reset接口
└── 成功：完整闭环 + 可追溯证据
    ├── AI推荐 → 规则检查 → 选课/候补 → 重算 → 追溯
    └── 每个环节都有真实调用和记录
```
